Per-app scoped mappings

One trigger, many behaviors — let a single key chord resolve to a different action depending on which app is frontmost, without breaking a single existing config file.

1 · What we're building

Today every mapping is global: Caps+J does the same thing in every app. We want a single trigger to branch by frontmost application — Caps+J is "down" everywhere, but "next browser tab" in Safari and "tmux switch pane" in iTerm2. This is conditional selection, not a filter: the same key produces genuinely different actions in different contexts.

The design rests on one fact about this app: a trigger is stable and unique. A key chord is one trigger. So instead of scattering scoped copies of a trigger across the config, we keep the trigger as the top-level identity and hang an ordered list of conditional bindings underneath it.

Core idea

A mapping = one stable trigger + a single default action + an ordered list of bindings, where each binding is { when: [conditions], action }. When the trigger fires, bindings are evaluated top-to-bottom; the first whose conditions match wins; otherwise the default action runs; otherwise the key is swallowed as a no-op.

2 · Two stages: selection vs execution

The whole feature lives in one of two independent stages. A key event first selects which action to run, then executes it. Per-app scope only changes selection. Execution — including how held modifiers (Option/Control/Cmd) are forwarded into the synthesized keystroke — is untouched.

Figure 1. Scope is purely a Stage-1 concern. The Stage-2 modifier-forwarding policy (whether a held Option rides along into the produced key) is independent and tracked separately in issue #5 — it is not part of this work.

One consequence worth recording: because a trigger can now resolve to different action kinds per app, the existing "modifier forwarding depends on action kind" rule becomes more visible. Caps+Option+J may forward Option in Finder (a directional action) but drop it in Safari (a keyCombo). That is pre-existing behavior, not a regression introduced here — see issue #5.

3 · Disk format & downgrade safety

The hard constraint: an older build keys the entire config by trigger — it dedupes mappings by trigger, finds entries by trigger, and keys its lossless-preservation tables by trigger. Any duplicate-trigger entry that lands in mappings is destroyed when an old build saves. So the format must keep mappings trigger-unique and put everything new where old builds will preserve-but-ignore it.

The grouping model satisfies this for free. The default action stays in the existing action_id (or inline action) field — exactly where old builds read it. The conditional bindings go in a new per-mapping bindings key, which old builds treat as an unknown per-mapping key and round-trip verbatim through the existing preservation machinery.

# action_mappings.yml
actions:
  next-tab:
    name: "Next browser tab"
    action: { kind: key_combo, target_key: 30, with_cmd: true }

mappings:
  - trigger: { kind: hyper_plus_key, key: 38, with_shift: false }
    action_id: builtin.down            # ← DEFAULT. Old builds read this.
    bindings:                          # ← NEW. Ordered. Old builds preserve + ignore.
      - when: [ { type: frontmost_app, include: [com.apple.Safari, com.google.Chrome] } ]
        action: next-tab
      - when: [ { type: frontmost_app, include: [com.googlecode.iterm2] } ]
        action_id: builtin.down        # bindings reference actions the same way mappings do
Figure 2. Because mappings stays trigger-unique, the old build's dedupe never fires and bindings survives a save untouched. A user who never adds an app rule produces a file byte-identical to today's.

Implementation note: add "bindings" to ConfigStore.mappingKnownKeys in the new build. This isn't about double-emission (mergeUnknownKeys only re-adds a key the fresh encode omitted, so a freshly-encoded bindings is never duplicated). It's about avoiding zombie data: if bindings stays "unknown" and a user clears all app rules, the fresh encode drops the key, the preserved stale node merges back, and the deleted rules resurrect. Marking it known stops that. Old builds — which lack the entry — keep treating bindings as unknown and preserve it verbatim; that asymmetry is exactly what we want.

4 · In-memory model

On disk the default lives in action_id and the rest in bindings, but in memory it is one uniform structure. ActionMappingEntry gains a bindings array; each Binding mirrors the existing action-reference duality (id preferred, inline fallback) and carries an ordered when list of typed conditions.

Figure 3. No per-binding id. Conflict resolution is positional order (saved verbatim), the UI rewrites the whole array on every edit, and the lossless guarantee stays at mapping granularity — so a stable binding identity buys nothing.

Why no binding id: the only scenario that would need one is preserving a future version's unknown sub-fields inside a binding across a user reorder. That extends the lossless contract below mapping level, which we explicitly do not promise (consistent with today). If that ever changes, revisit.

5 · Resolution algorithm

Selection is two nested steps: pick the trigger group (this is where the existing shift-fallback lives), then pick a binding within it by current context. Conflict resolution within a group is pure declaration order — first match wins.

Figure 4. "No group" means the trigger isn't mapped at all → the key passes through normally (today's behavior). "Group matched but nothing applies" → swallow as a no-op, because the user did claim this chord — note this is a deliberate change from today, where handleCapsRemap passes the key through when a matched mapping has no resolvable action.

The shift-fallback interaction generalizes cleanly. When Caps+Shift+J has no group of its own, fall back to the Caps+J group, resolve its effective action under the current context, and apply allowShiftFallback to that resolved action. Group-selection happens first; binding-selection second.

// ActionExecutor — sketch (Engine thread; reads cached context)
static func effectiveAction(_ entry: ActionMappingEntry, _ ctx: RuntimeContext) -> ActionConfig? {
    for b in entry.bindings {                 // declaration order = priority
        if conditionsSatisfied(b.when, ctx) {     // unknown condition type ⇒ false
            if let a = ActionsRegistry.shared.resolve(b) { return a }
        }
    }
    return ActionsRegistry.shared.resolve(entry)  // default; nil ⇒ caller no-ops
}

6 · Frontmost-app tracking & threading

The CGEventTap callback runs on a dedicated CFRunLoop thread, not the main thread. Calling NSWorkspace.shared.frontmostApplication there is unsafe and can block the hot path. So the frontmost bundle id is cached: a main-thread observer updates it on app-activation notifications; the engine thread only reads the cached value behind a lock — the same shape as EngineState.

Figure 5. The hot path never touches AppKit. A few-millisecond lag right after an app switch is acceptable: at worst one keystroke uses the previous app's scope.

8 · UI changes

The mapping editor grows from "pick one action" to "pick a default action + manage an ordered list of app rules." Each rule reuses the existing app picker (the NSOpenPanel-based chooseApp flow already used by the open_app action, which yields a bundle_id + display name and resolves an icon).

  • AddEditMappingView — add a "Per-app rules" section: the existing action picker becomes the default; below it, an add/remove/drag-reorder list where each row = an app picker (→ include list) + an action picker. Validate each row has ≥1 app and a resolvable action.
  • MappingsCard — show a per-row badge when a mapping has bindings (e.g. +2 app rules); keep the existing invalid-orange treatment for unresolvable actions, including those inside bindings.
  • L10n.swift — add keys for the new labels across en / zh / ja / de (e.g. mappings.default_action, mappings.app_rules, mappings.add_app_rule, mappings.applies_in, overlap-warning toast).

9 · Migration

No active migration step. Existing structured configs and the legacy 2.0 bare-list both load as entries with a default action and an empty bindings array. The file is rewritten with a bindings key only once the user actually adds an app rule — until then it stays byte-identical to today's output.

10 · Testing

  • Resolution unit tests: order priority; first-match-wins; default fallback; no-op when neither matches; pass-through when trigger unmapped; shift-fallback resolves through bindings; unknown condition type ⇒ skipped.
  • Codable + lossless round-trip: encode/decode of bindings; case-insensitive bundle-id match; downgrade simulation — load a file with bindings using the old known-keys set, save, and assert bindings is preserved verbatim and no mapping is dropped.
  • Manual: the §1 example (Finder/Safari/iTerm2) plus an app-switch latency sanity check.
  • Per project rule: a Codex review of the touched files (Model + Engine + UI) before declaring done.
