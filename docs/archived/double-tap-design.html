Double-tap pre-empt design

How HyperCapslock hides the 310 ms input lag that double-tap support introduced — without giving up double-tap detection — by treating the user's next keystroke as a third disambiguation signal.

1 · Why there is a delay at all

Commit 9b60ab7 added a DoubleTapHyper trigger so users can bind any action — Spotlight, Raycast, a shell command — to two quick taps of the Hyper key (CapsLock remapped to F18 via hidutil). The state machine in handle_short_tap needs to disambiguate two physically identical events:

  • Single tap — toggle CapsLock immediately.
  • First half of a double tap — do not toggle; wait to see if a 2nd tap arrives within DOUBLE_TAP_WINDOW_MS (300 ms), in which case fire the configured action and suppress the toggle entirely.

The only way to tell them apart is to wait. So when a DoubleTapHyper mapping exists, the toggle is deferred by DOUBLE_TAP_WINDOW_MS + 10 = 310 ms via a spawned thread that calls compare_exchange on a shared timestamp.

Figure 1. The 310 ms defer is invisible if the user does nothing, but corrupts the first character if they immediately start typing.

2 · The insight

The defer exists for one reason: we don't know whether a 2nd F18 tap is coming. But we don't need to only watch for a 2nd F18 tap — anything that rules out a 2nd tap is also a useful signal.

And there's an obvious one: any non-F18 KeyDown during the window. Humans cannot physically interleave a printable letter between two intended Caps taps in under 300 ms. If a letter arrives, the pending tap was definitively a single tap, and we can fire the toggle right then — synchronously, inside the same event-tap callback that's about to deliver the typed key.

Core idea

The event we're about to deliver is itself the proof we needed. Use it as a third disambiguation signal alongside the F18 KeyDown and the timeout.

Figure 2. The pre-empt path collapses the visible lag to whatever delay exists between the user's tap and their next keystroke — typically tens of milliseconds.

3 · The three load-bearing parts

The change is small (one block, ~15 lines of code), but every line is doing real work. Removing any of the three pieces below re-introduces a specific bug.

3.1 · The atomic protocol

Two threads can race to fire the same pending toggle: the spawned timer (sleeping 310 ms) and the event-tap callback (responding to a keypress). Both must be able to claim the pending tap, and exactly one must fire the toggle.

The token is a single AtomicU64:

static LAST_TAP_AT_MS: AtomicU64 = AtomicU64::new(0);
// 0 = no pending tap. Otherwise: the timestamp the tap landed.

The timer claims with compare_exchange(scheduled_for, 0) — succeeds only if nobody else has touched it. The pre-empt block claims with swap(0) — unconditional take. Whichever runs first wins; the loser becomes a no-op.

Outcomes: swap-first → CAS sees 0 ≠ scheduled_for → CAS fails → timer skips. CAS-first → swap returns 0 → pre-empt skips. Exactly one toggle.
Figure 3. Lock-free claim protocol. Both branches use SeqCst ordering — overkill for this token, but matches the rest of hook_macos.rs for consistency.

3.2 · The CAPS_DOWN guard

Naïvely, "any non-F18 KeyDown" includes the chord key in a Caps+H sequence — and that's a bug. Consider: the user single-taps Caps (deferring a toggle), then immediately presses-and-holds Caps for a chord. While Caps is held, they press H. The H KeyDown would otherwise pre-empt the pending toggle, so:

  • H is interpreted via handle_caps_remap → cursor moves left ✓
  • and CapsLock toggles as a side effect of the chord ✗

The fix: skip the pre-empt when CAPS_DOWN is currently true.

Figure 4. Without the !CAPS_DOWN guard, scenario B would fire both the chord action and a CapsLock toggle — visible LED flicker, unexpected case shift on subsequent typing.

3.3 · The in-flight AlphaShift patch

This is the most counter-intuitive part of the fix and the part I had to verify empirically with the user.

When the pre-empt block calls toggle_caps_lock(), IOKit synchronously flips the system's CapsLock state. But the CGEvent the user just pressed — the one we're holding in our callback parameter — was created by the WindowServer before our callback received it, with the old modifier flags stamped onto it. The character translation downstream doesn't recompute the case from the live IOKit state; it reads CGEventFlagAlphaShift off the event itself.

So a synchronous toggle is not enough. We also have to flip the AlphaShift bit on the in-flight event:

let patched = event.get_flags() ^ CGEventFlags::CGEventFlagAlphaShift;
event.set_flags(patched);

XOR works for both directions: if CapsLock was OFF, the event has no AlphaShift bit, XOR sets it; if CapsLock was ON, the event has the bit, XOR clears it. In both cases the patched flag matches the new IOKit state.

Figure 5. XOR is symmetric: it sets the bit if absent, clears it if present. The patched flag always matches the post-toggle IOKit state, regardless of direction.

6 · The actual code

// hook_macos.rs, inside the CGEventTap closure
if event_type_matches(event_type, CGEventType::KeyDown)
    && !CAPS_DOWN.load(Ordering::SeqCst)
{
    let pending = LAST_TAP_AT_MS.swap(0, Ordering::SeqCst);
    if pending > 0 && !IS_PAUSED.load(Ordering::SeqCst) {
        let age_ms = now_millis().saturating_sub(pending);
        let old_flags = event.get_flags();
        let patched = old_flags ^ CGEventFlags::CGEventFlagAlphaShift;
        log_macos(
            "INFO",
            &format!(
                "Caps(F18) pending toggle pre-empted by next keypress: \
                 keycode={} age={}ms; toggling CapsLock synchronously \
                 and flipping in-flight AlphaShift flag (0x{:x} -> 0x{:x}).",
                keycode, age_ms, old_flags.bits(), patched.bits()
            ),
        );
        toggle_caps_lock();
        event.set_flags(patched);
    }
}

7 · What would break if you removed each piece

Remove…SymptomSeverity
!CAPS_DOWN guard Caps+H chord toggles CapsLock as a side effect when performed within 300 ms of a prior tap. Visible LED flicker, subsequent typing in wrong case. HIGH
swap(0) on LAST_TAP_AT_MS Timer fires its toggle ~310 ms later anyway → double toggle → CapsLock flips back to original state. Net: nothing seems to change after a tap. HIGH
event.set_flags(flags ^ AlphaShift) Toggle fires synchronously, so chars 2..N have the new case. But char 1 was stamped with old flags → comes out in old case. "abc" prints as "aBC". MEDIUM
IS_PAUSED check If the user pauses HyperCapslock during the 300 ms window, a stale tap can fire a surprise toggle even though they explicitly stopped the app. MEDIUM
Whole pre-empt block Original 0.8.9 regression returns: 310 ms perceived input lag on every single-tap CapsLock when a DoubleTapHyper mapping is configured. HIGH

8 · Verification

Functional verification was empirical, performed by the user on macOS 14+:

  • Single tap → type "abc" with CapsLock starting OFF → "ABC" (was "aBC" before the AlphaShift patch was added; was "abc" before the whole pre-empt fix).
  • Single tap → type "ABC" with CapsLock starting ON → "abc".
  • Double tap → Spotlight opens (existing behavior, no regression).
  • Caps+H chord → cursor moves left, no incidental CapsLock flip (CAPS_DOWN guard verified).

The runtime log line includes keycode, age, and old/new flag bits in hex, so any future regression is diagnosable from /tmp/hypercapslock-macos.log without code changes:

[HYPERCAPS][macOS][1778594821][INFO] Caps(F18) pending toggle pre-empted
  by next keypress: keycode=66 age=312ms; toggling CapsLock synchronously
  and flipping in-flight AlphaShift flag (0x100 -> 0x10100).

9 · Alternative considered

Optimistic toggle — toggle CapsLock immediately on every tap, then undo on the 2nd tap if a double-tap is detected. Simpler state machine, no in-flight flag patching needed. REJECTED — the CapsLock LED briefly flashes on then off during every double-tap, ~150–200 ms visible flicker. The user explicitly preferred the pre-empt approach during design discussion.

Design conversation

The pre-empt approach was the user's idea, not mine. My initial proposal was a multiple-choice menu of optimistic-toggle vs shorten-window vs accept-the-lag. The user replied: "my suggestion: keep the delay, but if the hyper key was released, cancel the delay immediately." That triggered the realization that the next keypress could itself be the cancellation signal — which became this fix.
